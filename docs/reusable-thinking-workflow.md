# 可复用思考工作流：把 LLM 的推演沉淀为长期资产

> 目的：LLM 上下文有限，复杂设计不能只存在于对话里。
> 本文把 BestQ-A 当前已经形成的设计习惯，收束成一套**可重复执行的文档化工作流**，让后续 agent 和人类都能沿同一条思考链继续推进，而不是每轮从头发明。

---

## 0. 一句话原则

**不要把关键思考留在会消失的上下文里，要把它外化成可审计、可接力、可复用的文档对象。**

---

## 1. 为什么必须文档化

### 1.1 LLM 的上下文是易失的

对话再长，也会被：

- 截断
- 压缩
- 遗忘
- 换模型 / 换 agent / 换会话

所以如果关键推理只活在对话里，系统实际上没有积累任何长期资产。

### 1.2 复杂系统不能依赖“这轮刚好想明白了”

BestQ-A 的难点不是写一个函数，而是：

- 哲学层是否自洽
- 目标 contract 是否清晰
- 现有代码映射是否准确
- 实施切片是否可执行
- 自我批判是否被保留

这些都不是一次回答能稳定持有的。

### 1.3 文档化的真正目标

文档化不是为了“留档”，而是为了让思考过程变成：

- 可接力
- 可验证
- 可回滚
- 可批判
- 可版本化

---

## 2. BestQ-A 当前已经形成的思考链

根据当前 `docs/` 的真实演化，BestQ-A 已经形成了一个很明确的思考链：

```text
哲学层
  → 目标态合同
  → 现有代码映射
  → 实施切片
  → 审计与回归
  → 历史归档
```

对应到当前工作区中的文档层级：

| 层级 | 典型文档 | 作用 |
|---|---|---|
| 哲学层 | `epistemic-axioms.md` | 定义根公理、为什么系统必须这样设计 |
| 未决问题层 | `epistemic-open-questions.md` | 在写稳定 contract 前，把模糊问题磨锋利 |
| 目标态合同层 | `docs/current/*.md` 中的 draft contract | 把哲学变成工程必须服从的 schema / invariants / gates |
| 现状映射层 | `contract-vs-impl-audit.md`、`memory-layer-current.md` | 说明“代码现在到底是什么” |
| 实施计划层 | `.omx/plans/*.md` | 把 target contract 拆成可执行切片 |
| 历史层 | `docs/design_history/*.md` | 记录设计代际演进，不覆盖旧版本 |

---

## 3. 五种必须写出来的文档对象

如果后续 agent 想复用思考过程，至少要把以下五类对象文档化。

### 3.1 哲学文档（axioms）

回答：

- 系统究竟是什么
- 为什么不接受某些看起来更简单的方案
- 哪些原则是不可妥协的

适合写在：

- `docs/epistemic-axioms.md`
- 或其它根层哲学文件

**什么时候写**

- 当你发现自己在反复解释“为什么不是普通 RAG / 普通 QA / 普通 agent”时
- 当你需要给后续所有 contract 一个上游宪法时

### 3.2 未决问题文档（open questions）

回答：

- 哪些关键问题还没钉死
- 上一版哪里含糊
- 错了会怎样
- 候选答案有哪些
- 下一轮如何裁决

适合写在：

- `docs/epistemic-open-questions.md`

**什么时候写**

- 当问题还没足够清楚，不该直接进 contract 时

### 3.3 合同文档（contract）

回答：

- 目标对象长什么样
- 不变量是什么
- 生命周期是什么
- 哪些字段必须存在
- 哪些升级/门控条件必须满足

适合写在：

- `docs/current/*.md`

**什么时候写**

- 当一个概念已经不该继续停留在哲学层，必须进入工程约束时

### 3.4 实施计划文档（implementation plan）

回答：

- 先改哪里
- 先不改哪里
- 哪个切片最稳
- 改动影响哪些文件
- 验收标准是什么

适合写在：

- `.omx/plans/*.md`

**什么时候写**

- 当 target contract 已足够清晰，开始进入实际改代码阶段

### 3.5 审计文档（audit）

回答：

- 现在代码与合同之间差多少
- 哪些 claim 是假的
- 哪些地方名义上接了 contract，实际上没 enforce

适合写在：

- `docs/current/*-audit*.md`
- review findings 文档

**什么时候写**

- 当实现已经开始偏离合同，或需要对“我们已经做到了什么”进行反证时

## 4. 推荐的思考输出顺序

不要一上来就写 contract，也不要一上来就改代码。

更稳的顺序是：

### Step 1：先写哲学判断

形式：

- 这个系统到底是什么？
- 我拒绝什么？
- 哪条原则是根公理？

输出：

- 哲学文档 / 根判断

### Step 2：把模糊点单独列出来

形式：

- 问题的精确陈述
- 旧答案哪里含糊
- 错了会怎样
- 候选答案有哪些

输出：

- open questions 文档

### Step 3：再把已钉死的部分写成 contract

形式：

- schema
- invariants
- lifecycle
- gate

输出：

- current/ 下的 contract

### Step 4：做“现状 vs 目标”映射

形式：

- 当前代码最接近哪个对象
- 已有多少
- 缺口在哪里

输出：

- audit / current mapping 文档

### Step 5：再拆实施切片

形式：

- 哪一刀最稳
- 哪一刀最值
- 先改哪个模块

输出：

- `.omx/plans/*.md`

### Step 6：实现后回写审计和历史

形式：

- 哪些 contract 已部分落地
- 哪些目标态仍未实现
- 这一轮设计相对上一代升级了什么

输出：

- audit 更新
- design_history 新版本

---

## 5. BestQ-A 推荐采用的固定模板

后续大设计问题，建议默认都沿用下面这条模板，不要每轮重新组织思路：

```text
1. 先写一句根判断
2. 再写这次真正解决了什么
3. 再写还缺什么
4. 再写新的对象边界
5. 再写和现有代码的映射
6. 再写第一刀实施切片
7. 最后写自我批判和风险
```

这条模板的价值在于：

- 它强迫思考从抽象落到对象
- 它强迫设计和现有代码对齐
- 它强迫每一轮都有自我批判，而不是只讲新方案

---

## 6. 自我批判必须单独保留

一个高风险错误是：只记录最终方案，不记录为什么犹豫、为什么否定其它方案。

这会导致后续 agent：

- 重新讨论已经被否过的问题
- 不知道哪些边界是刻意保守的
- 误以为当前方案是“天然正确”，而不是“在多种坏方案里筛出来的”

所以每轮重要思考都应至少保留：

- 这轮最危险的误判是什么
- 为什么当前方案只是“阶段最稳”，不是“终局最优”
- 哪些地方是刻意延后，而不是忘了做

可以放在：

- `epistemic-open-questions.md`
- 实施计划中的 `Risks and Mitigations`
- 审计文档中的“总体结论”

---

## 7. 何时写到 `design_history/`

不是每次想法变化都需要升一个设计版本。

适合写进 `design_history/` 的情况：

- 系统的主隐喻变了
- 顶层对象集变了
- 根组织方式变了
- 当前版本需要被正式 supersede

不适合直接升设计版本的情况：

- 某个 contract 字段改动
- 某个实现切片更稳了
- 某个 gate 条件从 2 改到 3

也就是说：

**只有代际跃迁，才进 design_history。**

---

## 8. 何时写到 `current/`

适合写进 `docs/current/` 的内容：

- 已经足够清晰，可以约束代码
- 需要被 audit
- 需要被其它合同引用
- 是“目标态 schema / invariant / gate”

不适合写进 `current/` 的内容：

- 还在发散的问题
- 纯粹的感想
- 还没决定是否采用的对象模型

一句话：

**模糊的问题进 open questions，清晰的约束进 current。**

---

## 9. 何时写到 `.omx/plans/`

适合写进 `.omx/plans/` 的内容：

- 这轮要动哪些文件
- 先切哪一刀
- 验收怎么做
- 风险怎么控

不适合写进 `.omx/plans/` 的内容：

- 长期哲学判断
- 目标态 schema 本体
- 代际设计史

一句话：

**plan 负责施工，不负责立法。**

---

## 10. 推荐的最小文档化闭环

以后每当出现一个足够大的新主题，至少产出这 4 份：

1. `哲学判断`
2. `open questions`
3. `contract`
4. `implementation plan`

如果实现开始偏了，再补第 5 份：

5. `audit`

如果设计代际变化明显，再补第 6 份：

6. `design_history`

---

## 11. 对未来 agent 的操作指令

如果你是后续接手的 agent，默认按以下顺序读：

1. 先读 `docs/epistemic-axioms.md`
2. 再读相关 `docs/current/*.md`
3. 再读相关 `epistemic-open-questions.md`
4. 再读 `.omx/plans/*.md`
5. 最后才去看代码

理由：

- 先知道系统为什么存在
- 再知道目标态是什么
- 再知道哪里还没钉死
- 再知道这一轮要怎么施工
- 然后再看代码，才不会把当前实现误当成最终设计

---

## 12. 最终建议

BestQ-A 后续所有重要思考，都应该默认问自己四个问题：

1. 这是哲学判断，还是工程约束？
2. 这应该写进 `open questions`，还是 `current contract`？
3. 这只是实现切片，还是已经足以进入 `design_history`？
4. 这轮自我批判有没有被保留下来？

如果这四个问题回答清楚，思考就不会只停留在上下文里，而会真正变成可复用资产。

---

## 一句话收尾

**让系统变强的不只是更聪明的推理，而是把推理本身外化成下一轮还能继续利用的结构。**
