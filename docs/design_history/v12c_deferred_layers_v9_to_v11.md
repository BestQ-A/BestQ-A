# 递归式问答系统设计史 v12c：在当前能力边界下延后的 v9-v11 地平线层

## 1. 文档定位

本文件说明为什么 `v9`、`v10`、`v11` 现在被**有意延后**，以及这种延后为什么不是否定，而是按进入条件管理的架构顺序。

一句话：

```text
v9-v11 不是被放弃，而是被定义为需要更高实现前提、验证环境和制度约束后才能进入主线的地平线层。
```

---

## 2. 为什么要显式延后 v9-v11

根据当前路线图：

- `v9` 为零代码实现
- `v10` 为零代码实现
- `v11` 为零代码实现

这不是说这些版本不重要，而是说它们依赖的世界条件还没有齐。

当前主线刚开始把 v8 的可校准子集压入合同与执行桥；在这种阶段，如果再把 v9-v11 一并写成当前架构层，就会出现三类问题：

1. 把“远期组织原则”误写成“当前系统能力”
2. 把“缺验证环境的对象”误写成“已可运行机制”
3. 把“制度与文明级约束”误写成“文档中的口号层”

因此延后不是后退，而是防止设计史跨越现实。

---

## 3. 延后原则：地平线层，而非废案层

v12c 对 v9-v11 的态度是：

- 它们定义系统长期上限
- 它们不应被从设计史中删除
- 它们当前不进入主线能力叙述
- 它们必须各自拥有可检查的进入条件

因此最准确的定义是：

```text
v9-v11 = horizon layers
```

也就是：

- 当前用来约束方向
- 未来在条件满足时逐层吸收
- 在条件未满足前，不提前宣称已实现

---

## 4. v9：本体联邦层为什么延后

### 4.1 v9 要解决的不是“更多对象”，而是联邦现实

v9 的核心不是加几个类型名，而是要求系统真正能够处理：

- 多个局部 `OntologyModel`
- 显式 `PerspectiveModel`
- 合法 `TranslationFunctor`
- `ConflictSet` 与 `UnresolvedBoundary`
- 共享内核编译门控

这要求系统不只是有单一本体中的预测闭环，还要有跨本体的翻译验证与冲突治理。

### 4.2 当前为什么不能把 v9 写成主线能力

当前文档给出的现实约束非常明确：

- 路线图把 v9 标成零代码实现
- v9 所需测试环境是多智能体环境，如 Mineflayer / ROS2 + Gazebo
- 当前 `docs/current` 中并没有 `OntologyModel`、`TranslationFunctor`、`ConflictSet` 的 `current` 合同链
- 连 v8 的程序执行与反事实校准都还未转为稳定现实，跨本体联邦更无从成立

因此当前没有足够依据声称系统已经拥有：

- 多本体并存的运行主链
- 可验证的跨本体翻译
- loss profile 驱动的共享内核门控
- 不可通约边界的持续治理

### 4.3 v9 的进入条件

v9 不是“以后也许会做”，而是有明确进入门槛：

1. 单一本体内的 v8 最小预测闭环必须先成立

- `MechanismProgram` 不再只是草案对象
- `CounterfactualScenario` 不再只是占位轨迹
- `ExperimentDesign` 能稳定引用可区分的候选实验

2. 至少存在两个可持续维护的局部本体运行实例

- 各自有独立 Episode 采样
- 各自有稳定对象边界
- 不是同一套语言的简单别名

3. 必须存在可验证的跨本体验证环境

- 至少能复现“同一事件，不同观察”的场景
- 能显式输出翻译损失，而不是只输出成功映射
- 能保留不可翻译部分

4. 必须存在翻译合法性的审计入口

- `preservedInvariants` 可检查
- `lossProfile` 可落盘
- `ConflictSet` 可持续保留，而非自动抹平

只有当这些条件成立时，v9 才从“联邦理念”转为“可实现层”。

---

## 5. v10：观察者 / 部署自反层为什么延后

### 5.1 v10 的实质，是把系统自己拉回世界内部

v10 要引入的不是装饰性 metadata，而是整套自反约束：

- `ObserverModel`
- `InstrumentModel`
- 带位置的 `ObservationRecord`
- `DeploymentShiftModel`
- 制度化 compile 与 `NormativeKernel`

它要求系统承认：

- 谁在看会影响看到什么
- 用什么仪器会影响观测质量
- 一旦部署，系统会改写后续世界分布
- compile 不再只是模型内部动作，而是制度动作

### 5.2 当前为什么不能把 v10 写成主线能力

当前虽然 `ObservationModel` 已进入 `current`，但这并不等于 v10 已具备。

原因在于：

- `ObservationModel` 只是“从状态到观测的投影规则”，并不等于完整 observer / instrument 层
- 路线图把 v10 标成零代码实现
- v10 所需验证环境是可逆部署环境，如 Kubernetes 金丝雀或 LLM A/B 沙盒
- 当前没有证据表明主线已经稳定建模 deployment-induced shift
- 当前也没有制度化 compile 对象链进入 `docs/current`

因此现在最多只能说：

- 主线吸收了 v10 的一个前置台阶，即“观测不是事实本身”
- 但尚未进入“观察者、仪器、部署、制度共同构成闭环”的阶段

### 5.3 v10 的进入条件

v10 成为可实现层，至少需要以下条件：

1. observer / instrument 不能只停留在概念层

- `ObserverModel` 与 `InstrumentModel` 必须成为显式对象
- 观测记录必须能稳定回指观察者与仪器
- 盲区、偏差、校准状态不能只是注释性文本

2. 部署必须被当作可回放干预来记录

- 至少有一个真实系统能复现“部署改变了后续分布”
- `DeploymentShiftModel` 必须能回指具体部署动作与失效假设
- 系统必须能区分外部世界变化与自诱发漂移

3. 必须有可逆部署测试床

- 能上线
- 能观测
- 能回滚
- 能把回滚与重新校准写回主链

4. compile 必须具备最小制度化门控

- proposal / challenge / replay / approval 不再只是设计措辞
- 至少存在最小审理协议的对象化记录
- 高风险域写入必须可阻断，而不是事后补记

这些条件满足后，v10 才不是“系统承认自己在世界内部”的哲学表述，而是可执行的架构层。

---

## 6. v11：文明记忆 / 失败边界层为什么延后

### 6.1 v11 的实质，是把长期求真能力本身作为保存对象

v11 要保存的不只是答案，也不只是机制，而是：

- `ProofLineage`
- `CounterexampleCommons`
- `FailureBoundaryArchive`
- 宪法层历史
- 未解决前沿

它要回答的是：

- 哪些失败定义了生存边界
- 哪些反例不能被静默删除
- 哪些机制为什么被回滚
- 哪些冲突必须为未来保留

### 6.2 当前为什么不能把 v11 写成主线能力

当前没有足够现实基础把 v11 写成可运行层，原因很直接：

- 路线图把 v11 标成零代码实现
- v11 所需验证环境包括红队审计、长期演化实验、灾难压力测试
- 当前主线连 v8 的完整预测误差校准链都未稳定建立
- 如果基础 replay、counterexample、deployment shift 都还未成熟，文明级 archive 只能是口号，不能是制度现实

因此 v11 当前只能作为高阶约束方向，而不能作为当前能力陈述。

### 6.3 v11 的进入条件

v11 要从修辞层变成实现层，至少需要：

1. 前置层稳定

- v8 的预测 / 实验 / 结果校准链已经可持续运行
- v9 的本体冲突与翻译损失能够被制度化保留
- v10 的部署漂移与观察者位置已经可记录、可回放、可审计

2. 失败边界必须有结构化来源

- 失败不是日志碎片，而是可聚合对象
- 至少能稳定形成 `ActionExecution → OutcomeRecord → counterexample / failure` 的链
- 机制失效面与 broken invariants 需要可追踪血统

3. 必须有长期审计与红队基础设施

- 能测试“反例压制”“历史抹除”“静默统一”“权限绕过”等攻击
- 这些攻击的拦截记录必须可保留、可回放、可追责

4. 必须有跨代记忆的真实维护需求

- 系统运行时间和知识更新频率足以暴露历史遗忘问题
- compile / rollback / challenge 已经成为现实操作，而不是未来想象
- failure archive 保存的是活边界，而不是概念样例

只有满足这些条件，v11 才能从“文明级比喻”进入“文明级基础设施”。

---

## 7. 三层延后的共同逻辑

v9-v11 被延后，不是因为它们不正确，而是因为它们都共享一个更高前提：

```text
必须先有可运行、可校准、可审计的局部闭环，
才能上升到跨本体、部署自反、文明记忆这些更高层。
```

具体地说：

- 没有 v8 的最小预测闭环，就没有 v9 的联邦翻译对象基础
- 没有 v9 的冲突治理，就没有 v10 的制度化 compile 现实
- 没有 v10 的部署与观察者自反，就没有 v11 的真实 failure boundary archive

因此延后顺序本身就是架构纪律。

---

## 8. 对 v9-v11 的最终判断

v12c 对 v9-v11 的判断是：

```text
它们不是当前主线层，
但它们定义了主线未来必须长成什么样。
```

更精确地说：

1. `v9` 是联邦地平线

- 主题：多本体并存、翻译法律、冲突保留、共享内核门控
- 进入条件：多本体运行实例、翻译验证环境、loss/conflict 审计成立

2. `v10` 是自反部署地平线

- 主题：观察者位置、仪器偏差、部署漂移、制度 compile
- 进入条件：observer/instrument 对象化、可逆部署测试床、制度门控成立

3. `v11` 是文明记忆地平线

- 主题：proof lineage、反例公地、失败边界、宪法性护栏
- 进入条件：长期审计、红队、回滚、失败血统保存都已成为现实需要

所以当前最准确的表述不是“v9-v11 还太远”，而是：

> **v9-v11 已被定义为有明确进入门槛的地平线层；在这些门槛满足之前，它们负责约束方向，而不负责制造虚假的当前能力叙事。**
